# ADR 0008: 雑談ベースの自然会話から思考ログを完成させる（Conversation-first Thought Log）

- **Status**: Accepted
- **Date**: 2026-01-11
- **Deciders**: Product / Engineering
- **Related**:
  - `docs/bmad/spec-current.md`（思考ログ: `#整理開始/#整理終了`、logs/sessions）
  - `docs/bmad/specs/04-thought-log-mode.md`
  - `docs/bmad/specs/03-line-session-and-command-routing.md`
  - `docs/bmad/gaps.md`（DeepSeek失敗時、ジョブ/認証など）

## Context
現状の思考ログは、ユーザーが `#整理開始` を明示して「logモード」に入り、対話を進め、`#整理終了` 時点で `logs` に1件保存する設計である。

一方でプロダクト方針として、返信体験は「コマンド操作」ではなく **雑談に見える自然会話** を維持したい。ただし内容は単なる日常会話ではなく、ユーザーが話しかける背景には「考えごと/悩み/迷い」がある前提を置き、エージェントが雑談ベースで読み取りながら問いを重ね、最終的に **ユーザーが欲しい答え（=自分の気づき/整理）に到達した時点で思考ログが完成**する体験を目指す。

この体験を導入するには、以下を仕様として決める必要がある。

- 自然会話を「雑談として返す」一方で、どの条件で `logs` に保存するか（自動保存/同意/明示コマンド）
- 既存の明示ログ（`#整理開始`）との関係（置き換え/併存/優先順位）
- 誤検知（雑談なのにログ化される等）・プライバシー・削除/訂正の導線
- `sessions`（全文）と `logs`（要約/構造化）の責務分離と参照関係
- 記憶（ユーザーを“正しく管理する”ための情報）をどの単位で保持し、ユーザーがどう制御できるか

## Decision
以下を採用する（提案）。

1) **Conversation-first（暗黙ログ）を追加し、明示ログ（`#整理開始`）と併存させる**
- 明示ログは「ユーザーが“残す/整理する”意思が明確」な強いモード。
- 暗黙ログは「雑談として進みつつ、気づきをログとして完成させる」体験。

2) **暗黙ログは“下書きの提示”までに留め、`logs` への保存はユーザーの明示コマンドでのみ行う**
- 雑談体験と心理的安全性（「勝手に記録される不安」）を最優先する。
- エージェントは、会話の区切りで「下書き（要約＋気づき＋結論/次の一歩）」を提示してよい。
- ただし **保存は自動でも確認でもなく**、ユーザーが保存コマンドを送った場合のみ実施する。
- 保存コマンド名はspecで固定する。
  - 採用: `#ログ保存`
  - 互換/alias（任意）: `#メモ保存`

3) **暗黙ログの“完成”は、エージェント側の推定ではなく「ユーザーの到達/合意シグナル」を条件にする**
- 例: ユーザーが「なるほど」「腹落ちした」「これだ」「じゃあこうする」等の確定表現を返す、またはエージェントが要約を返しユーザーが肯定する。
- この“完成判定”は、実装上はルール + LLM分類の併用を想定（最終仕様はspecで定義）。

4) **暗黙ログは `logs` に1件保存し、元の会話全文は `sessions` 参照で辿れるようにする**
- `logs.rawText` には「要約＋気づき＋結論（次の一歩）」を入れる。
- `logs.type`（既にギャップで触れている拡張）を用い、暗黙ログを識別可能にする（例: `insight` または `chat_insight`）。

5) **記憶はタイプを分け、ユーザーが制御できる形で保持する**
- **安定記憶（ユーザー明示で更新）**: 例）表示名、キャラクター、トーン、呼びかけ頻度、NG事項。保存先は `user_settings` を想定。
- **推論の芯（短文サマリ）**: 400文字程度を上限とし、「ゆいが一貫して振る舞うための運用ルール/扱い方」を保持する（例: 介入の強さ、励まし方、分割の好み等）。
  - 400文字は“全人格”を表現するには不足し得るため、役割を「インデックス（最重要の行動指針）」に限定する。
- **当日メモ（TTLあり）**: 例）今日の制約（時間/デバイス/エネルギー）や一時的な状況。日付が変われば原則リセット。

6) **下書きの保持方針（推奨）**
- 目的: 「保存し忘れの取りこぼし」を減らしつつ、「勝手に記録される不安」を増やさない。
- ルール:
  - **下書きはユーザーあたり常に1件だけ**（新しい下書きが生成されたら上書き）
  - **TTLは24時間**（期限切れは破棄）
  - 保存されるのは下書き（要約＋気づき＋結論/次の一歩）のみで、`logs` には入れない（`#ログ保存` のみで保存）
  - ユーザー操作:
    - `#下書き`: 現在の下書きを再表示
    - `#下書き破棄`: 下書きを破棄
    - `#ログ保存`: 現在の下書きを `logs` に保存（保存後は下書きを破棄）

## Options Considered
- Option A: **暗黙ログを導入しない（現状維持）**
  - 長所: 誤記録/プライバシー事故が起きにくい。実装が単純。
  - 短所: 雑談ベースの自然会話を「思考ログとして残る体験」に繋げにくく、ユーザー側のコマンド負担が残る。

- Option B: **暗黙ログを“常に自動保存”する（同意なし）**
  - 長所: ユーザーの手間が最小。取りこぼしが少ない。
  - 短所: 誤検知・過剰記録・プライバシー不安が最大化し、雑談体験の心理的安全性を損ねる恐れ。

- Option C: **暗黙ログを導入し、完成時にだけ保存確認を行う**
  - 長所: 雑談体験を維持しつつ、誤記録とプライバシー不安を抑えられる。
  - 短所: 「確認」自体がログ化を意識させ、雑談感を損ねる可能性がある。

- Option D: **暗黙ログを導入し、保存は明示コマンドのみ（本ADRの提案）**
  - 長所: 「勝手に記録されない」を保証でき、心理的安全性が最大化する。雑談のまま続けやすい。
  - 短所: 保存し忘れで取りこぼしが起きうる（下書き再提示/リマインド設計が必要）。

## Consequences
- **Positive**:
  - 雑談として自然に会話しながら、結果として思考ログが「完成」する体験を作れる。
  - 明示ログ（`#整理開始`）は残るため、従来ユーザー/既存運用との互換を維持できる。
  - 保存を明示コマンドに限定することで、誤記録やプライバシー懸念を最小化できる。
  - 記憶をタイプ分離することで、「対話パートナー」と「良き管理者」の両立（押し付けない介入）がしやすくなる。

- **Negative / Risks**:
  - “完成判定”の品質が低いと、下書き提示が多すぎる/少なすぎる問題が起きる。
  - 雑談に見せるためのプロンプト/応答設計が難しく、LLM失敗時のUXが悪化し得る（`gaps.md` のDeepSeek失敗時挙動と連動）。
  - 保存を明示にすると、取りこぼし（保存し忘れ）が起きうる。
  - 「下書き」の保持方針（破棄/期限付き/再提示）を誤るとUXが不安定になる。

- **Operational notes**（運用/移行/互換性/ロールバック）:
  - 初期は暗黙ログを「保守的（確認少なめ）」にし、取りこぼしより誤記録を避ける。
  - 明示ログは従来通り動作。
  - ロールバックは「暗黙ログの入口（下書き提示）を無効化」するだけで可能にする。
