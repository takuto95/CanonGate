## 統合仕様（協議用マスター要約）
このセクションは「今ある仕様を1枚で俯瞰できる」ように、現状仕様を要約したもの。詳細は以降の各章（B/M/A/D）および関連ドキュメントを参照。

### システム概要
- **ゴール**: LINE上で「思考ログ→整理（AI）→タスク化→日報→週次レビュー」を完結させる。
- **中核**: Next.js App Router（`app/api/**/route.ts`）が実運用の中心。永続化はGoogle Sheets、LLMはDeepSeek、UIはLINE（Flex/リッチメニュー）＋LIFF。

### データ（要点）
- **Goal**: `id,title,confidence,status,createdAt,updatedAt`（statusは主に `pending` 作成）
- **Chapter**: `id,goalId,title,order,status,hook,createdAt,updatedAt`
- **Task**: `id,goalId,chapterId*,lane*,chapterStage*,description,status(todo/done/miss),dueDate,priority(A/B/C),assignedAt,sourceLogId`
- **Log**: `id,timestamp,userId,rawText,emotion,coreIssue,currentGoal,todayTask,warning,(type*),(sourceSessionId*)`
- **UserSettings**: `userId,characterRole,messageTone,displayName,createdAt,updatedAt,(userProfileRole*),(constraints*),(analysisSummary*),(userMemorySummary*)`
  - `analysisSummary`: **AIが更新する分析サマリ（ユーザー編集不可）**
  - `userMemorySummary`: **AIが更新するユーザー記憶メモ（ユーザー編集不可 / 口調は変えず“内容”だけ特化させる）**
  - `constraints`: **制約（推定が基本 + 必要ならユーザーが上書き）**。上書きのTTLは「今日だけ」
- **Sheets**: `goals/chapters/tasks/logs/sessions/user_settings`

### 主要フロー（要点）
- **思考ログ（log）**: `#整理開始` → 対話（返信3分割） → `#整理終了`（logsに1件保存）→ 必要なら `#タスク整理` でタスク生成
- **日報（daily）**: `#日報開始` → `done <番号|taskId>` / `miss <番号|taskId> 理由` / `list` / `status <taskId>` → `#日報終了`（サマリー＋提案、必要なら後続タスク追加）
- **ステータス**: `#ステータス`（番号選択式メニュー）

### ジョブ/通知（要点）
- **morning**: `/api/jobs/morning` が「今日の焦点」をFlex MessageでPushし、`sessions` に `morning_order` を記録する。
- **night**: `/api/jobs/night` が夜の確認をPushし、返信（完了/未達）で `morning_order` のタスクを更新する。
- **weekly**: `/api/jobs/weekly` が週次レビューをPushする。
- **morning/nudge/night**: ジョブは用途別に分割して実行する（morning=朝通知, nudge=昼の促し, night=夜確認）。
- **cadence**: `/api/jobs/cadence` は互換/手動用（旧: 4時間枠の統合）。
- **daily-review**: `/api/jobs/daily-review` が日次で「タスク見直し案」を生成して提示する（**反映はボタンで確定**。自動適用はしない）。
  - **リマインド上限**: 日中の nudge（12/16/20枠）は **最大2回/日** まで送ってよい（黙っている理由は「忙しい/面倒/寝てる」を想定し、押しすぎない）
- **ジョブ実行基盤**:
  - GitHub Actions（`.github/workflows/jobs-cadence.yml`, `.github/workflows/jobs-weekly.yml`）で定期実行する
  - `vercel.json` の Cron は無効（`crons: []`）

### UI（要点）
- **朝の通知**: Flex Message（カード＋ボタン: 今すぐ開始/後で/変更）。
- **意図違いの修正**: 提示されたタスクが想定と違う場合は「🛠 文言修正」から文言の言い換え相談に入れる（ボタン→入力）
  - **ガイド**: タスク自体が違う→`🔄 変更` / 表現や条件が違う→`🛠 相談/修正`
- **返信（見やすさ）**: 頻出の結果通知（完了/未達、ステータスの「今日のタスク」など）はFlex Messageでカード表示し、視覚的に把握しやすくする。
- **リッチメニュー**: 「📋タスク一覧」はLIFFを開く。
- **LIFF ID**: `2008835190-F0ZpGhEt`
- **LIFF（ステータス）**: `https://liff.line.me/2008835190-F0ZpGhEt/status` でステータス画面を開く（同一LIFF内で画面切替）。
- **LIFFファースト方針**: 一覧/俯瞰（タスク一覧・ステータス）はLIFFを正とし、チャット返信は短い要約＋LIFF導線を基本とする。

## M (Market / Modeling)
### 想定ユーザー
- LINEで日々の状況/感情/進捗を送る個人ユーザー（現状は **単一ユーザー運用** を前提にし、`LINE_USER_ID` を対象ユーザーとして扱う）。
  - マルチユーザー対応が必要になった場合は別途協議し、ADRで方針決定する（ADR: `docs/adr/0002-single-user-scope.md`）

### ドメイン（データモデル）
- **Goal**: `id,title,confidence,status,createdAt,updatedAt`
  - status: `pending | approved | archived`（ただし現行API/実装では主に `pending` を作る）
  - **ゴールとタスクの連携**: タスクの `goalId` でゴールと紐付く
  - **進捗計算**: ゴールに紐づくタスクの完了数/総数で進捗率を計算
- **Chapter**: `id,goalId,title,order,status,hook,createdAt,updatedAt`
  - **位置づけ**: ゴール（クエストライン）配下の「章」。タスクは `chapterId` で章に紐付けられる
  - status（現行想定）: `active | completed`（互換のため文字列として扱う）
- **Task**: `id,goalId,chapterId*,description,status,dueDate,priority,assignedAt,sourceLogId`
  - status: `todo/done/miss` を主に使用（文字列で自由だが日報処理がこれを前提）
  - priority: `A/B/C` を想定
  - **goalId**: どのゴールに貢献するタスクかを示す
  - **chapterId***: どの章に属するタスクか（未設定は互換/既存運用）
  - **lane***: `main | side`（未設定は `main` 相当）。メイン進行を邪魔しないためにサイド枠を用意する
  - **chapterStage***: 同一章内の“段”（例: 1,2,3...）。同じ段=並列、段が進む=後続
- **Log**: `id,timestamp,userId,rawText,emotion,coreIssue,currentGoal,todayTask,warning,type*,sourceSessionId*`
- **UserSettings**: `userId,characterRole,messageTone,displayName,createdAt,updatedAt,userProfileRole*,constraints*,analysisSummary*,userMemorySummary*`
  - `constraints` の最低語彙（最小セット）: **時間 / 環境（デバイス） / エネルギー**
- **SessionEvent (sessionsシート)**: `sessionId,userId,type,content,timestamp,meta`
  - mode: `log | daily`（startイベントのmetaで保持）

### 永続化（Google Sheets スキーマ）
- **goals**: `id,title,confidence,status,createdAt,updatedAt`
- **chapters**: `id,goalId,title,order,status,hook,createdAt,updatedAt`
- **tasks**: `id,goalId,chapterId*,lane*,chapterStage*,description,status,dueDate,priority,assignedAt,sourceLogId,(reason*)`
- **logs**: `id,timestamp,userId,rawText,emotion,coreIssue,currentGoal,todayTask,warning,type*,sourceSessionId*`
- **sessions**: `sessionId,userId,type,content,timestamp,meta`
- **user_settings**: `userId,characterRole,messageTone,displayName,createdAt,updatedAt,userProfileRoleWork*,userProfileRolePrivate*,constraintsJson*,constraintsSource*,constraintsUpdatedAt*,analysisSummary*,analysisUpdatedAt*,analysisVersion*,userMemorySummary*,userMemoryUpdatedAt*,userMemoryVersion*`
  - focus（Option A）: `focusGoalId*`, `focusChapterId*`, `focusUpdatedAt*`
  - `constraintsJson*` の最低語彙（最小セット）:
    - `timeBudgetMinutes`: `5 | 15 | 25 | 60`
    - `device`: `mobile_only | pc_available`
    - `energy`: `low | medium | high`
    - （任意）`context`: `outside | home | office`
  - `constraintsSource*`: `inferred | user_override`
  - `constraintsUpdatedAt*`: 上書きは **「今日だけ」**（日付が変わったら推定に戻る）

### 主要フロー（LINE）
#### 1) 思考ログ（logモード）- **対話的な深層心理探求**
- 開始: `SESSION_START_KEYWORD`（デフォルト `#整理開始`）
  - 互換コマンド: `#ログ開始` でも開始できる（レガシーalias）
  - メッセージ: 「今、何が気になってる？ふわっとした気持ちでいい。そのまま送って。」
- **対話の特徴**:
  - ユーザーの言葉の奥にある深層心理を探る質問を返す
  - 表面的な言葉ではなく、「なぜ？」「本当は？」と問いかける
  - 思考の矛盾や前提を優しく指摘し、ユーザー自身が気づくように導く
- **メッセージ分割**:
  - やり取りが増えても読みやすいよう、AIの返信は3つに分割
  - 1つ目: 感情の共感
  - 2つ目: 現状の整理と気づきの提案
  - 3つ目: 核心を突く質問
- 会話: ユーザーが送るたびに DeepSeek で深層心理分析（JSON抽出）し、対話的に返信
- **コンテキスト参照**: 思考ログのDeepSeek入力には `user_settings.analysisSummary` / `user_settings.userMemorySummary` / `constraintsJson` などの「ユーザー固有コンテキスト（事実メモ）」を付与し、過去ログから抽出された要点も踏まえて深掘りする
- **長文化ガード**:
  - DeepSeekへ渡す思考ログは、長文時に「先頭抜粋 + 直近抜粋」に圧縮して入力上限を守る（全文は `sessions` に残る）
  - DeepSeek失敗時でもWebhook処理は落とさず、`#整理終了` でいったん保存する導線を提示する
  - **区切り提案**: 思考ログのやり取りが **3往復** くらい続いたら、返信カードに「🧾 ここまでで整理終了」ボタンを出し、ユーザーが任意で区切れるようにする（自動終了はしない）
- 終了: `SESSION_END_KEYWORD`（デフォルト `#整理終了`）
  - 互換コマンド: `#ログ終了` でも終了できる（レガシーalias）
  - メッセージ: 「思考の整理、お疲れ様。今の気持ちを言語化できたね。」
- **ログの独立蓄積**: `#整理終了` 時点で、思考ログの `logs` レコードを **必ず1件** 作成する（タスク化は別）
- その後: `TASK_SUMMARY_COMMAND`（デフォルト `#タスク整理`）で、終了済みログを基にタスク生成
  - **修正相談（意図違い対応）**: タスク化後に「🛠 修正相談」を押すと、該当ログ由来のタスク（最大3件）を提示し、`✏️ 修正` → 修正文入力（ボタン後のみ自由記述）で言い換えできる
  - **提示後の相談（今日のタスク等）**: 朝通知/ステータスで提示されたタスクについても「🛠 相談」から **文言/優先度/期限/ゴール** の修正ができる（ボタン中心、入力はボタン後のみ）
    - **期限入力の完全ボタン化**: 期限は `今日/明日/週末/来週末/なし` を基本のボタンで提供し、自由入力は `📝 その他（入力）` を押した後だけ受理する

#### 2) タスク整理（ログ→タスク生成）
- 入力: セッションのユーザー発話ログ（複数行の transcript）
- 出力: logs に1件追加 + tasks に複数件追加（AI JSONの `tasks` を採用）
- **紐付け**: タスク生成時は `tasks.sourceLogId` に元ログの `logId` を保存する
- **ゴール自動作成**: AIが抽出した `currentGoal` を自動的にgoalsシートに追加（重複チェック付き）
  - 同じタイトルのゴールが既に存在する場合は既存のIDを使用
  - 新規の場合は `g_` prefixのIDを生成してgoalsシートに追加
  - タスクの `goalId` にはgoalsシートの実際のIDを保存
 - **章（chapters）**: AIが章情報を返した場合、`chapters` に章を追加し、タスクに `chapterId` を割り当てる（返せない場合は `chapterId` 空で互換動作）
 - **lane/chapterStage（Option A）**:
   - タスク生成時に `lane`/`chapterStage` を埋められる場合は採用する（未設定は互換動作）
   - `lane=side` は「日報由来の残タスク」等、メイン進行を邪魔しない枠として扱う

#### 3) 日報（dailyモード）
- 開始: `DAILY_START_KEYWORD`（デフォルト `#日報開始`）
  - todoタスクを優先度順に **2-3件** 表示（見やすさ重視）
  - **各タスクにゴール情報を表示**（どのゴールに貢献しているか）
  - 全件表示は `list` コマンドで確認可能
  - `#日報開始 1,3` のように後ろに番号/IDを並べると、日報対象タスクを絞り込んで開始できる
- 更新（同モード中のメッセージ）:
  - **推奨入力形式**（動詞が先）:
    - 完了: `done 1` / `完了 1`
      - **done時の次タスク案内**: 残りtodoがある場合、次のタスクを自動で提示（「もう1件いける？」）
      - モチベーション向上: 日中に完了した場合もう一つタスクをこなせるよう案内
    - 未達: `miss 2 理由` / `未達 2 理由`
      - **miss時は次のアクション提案**（再挑戦/分割/延期の選択肢）
    - メモ: 上記以外は全てメモとして記録
  - **間違った形式の検知**:
    - 逆順（`1 done` / `2 miss 理由`）を送ると、正しい形式を提示してエラーを返す
    - 理由: タスクIDは `t_` から始まるため、番号とIDの混同を避けるため動詞を先に書く
  - task status 更新（done/miss）
    - **更新成否を検証し、失敗時は詳細なエラーメッセージを返す**
    - 更新後の状態を確認して整合性を保証
  - sessions に daily_update を記録
  - `list` / `一覧` で todo 一覧を再表示できる
  - `対象 1,3`（または `report 1,3`）で「日報対象タスク」をメッセージ指定できる（解除は `対象 全部`）
  - **番号**は todo全件リスト基準（対象で絞っても番号は変わらない）
  - `status <taskId>`（または `ステータス <taskId>` / `確認 <taskId>`）でタスクの現在の状態を確認できる
  - **メッセージは分割表示**で読みやすく（LINEのreplyMessages機能を使用）
- 終了: `DAILY_END_KEYWORD`（デフォルト `#日報終了`）
  - daily_update を集計してサマリー化
  - **モチベーション向上機能**:
    - 進捗サマリー表示（「今日は2件完了！達成率67%」）
    - ストリーク表示（「🔥 連続5日！」）- 日報を連続で記録した日数
    - **ゴール進捗表示**（「キャリアアップ: ████░░░░░░ 40%」）- 最大3件
  - サマリーを logs に追記（rawTextにサマリー文字列）
  - サマリーと未着手todo一覧を元に DeepSeek で「評価/明日の焦点/タスク見直し案/後続タスク」を生成し返信
  - 後続タスク（0〜5件）が提案された場合は tasks に todo として追加する（sourceLogId は日報logId）
  - タスク見直し案（再スケジュール案）を sessions に保存する
  - 再スケジュール案はこの時点では適用しない（提案表示のみ）
  - ユーザーが `DAILY_RESCHEDULE_COMMAND`（デフォルト `#再スケジュール作成`）を送ると、直近の日報提案から **再スケジュール用の新規タスク（todo）** を作成する
    - `#再スケジュール作成 <dailyLogId>` で対象日報を指定できる

### Cron/ジョブ
- `/api/jobs/morning` (GET/POST): 次の todo 1件を「今日の焦点」としてPush
  - **AIによるスマートタスク選定**: 期限、優先度、ゴールバランス、最近の進捗を考慮
  - 次のtodoの並び順（AI失敗時）: priority（A→B→C→不明）→ dueDate（早い順/空は後ろ）→ assignedAt（早い順/空は後ろ）→ シート行順
  - **モチベーション向上**: 日替わりの励ましメッセージ（「おはよう。今日もやっていこう。」など）
  - **パーソナライズ**: ユーザー設定に応じてキャラクターロール・トーンを適用
  - **リッチメッセージ化（Flex Message）**: 
    - 通知を見るだけでタスク内容が完全に把握できる
    - カード形式で表示（タスク名、ゴール、優先度、期限、AI選定理由、今日の傾向）
    - インタラクティブボタン付き（今すぐ開始/後で/変更）
  - **対話機能**: ボタンタップでアクション可能
  - Push本文は Flex Message 形式（従来のテキストから移行）
  - 当日の命令タスクIDを `sessions` に `morning_order` として記録する（todoが無い場合は空で記録し、前日の命令が誤って参照されないようにする）
- `/api/jobs/night` (GET/POST): 夜の確認メッセージをPush
  - 返信は `完了` または `未達 <理由1行>` のみを要求する
  - 返信が日報/思考ログのどのモードでもない場合でも、`完了/未達` を受理し、直近の `morning_order` に紐づくタスクを done/miss 更新（IDが取れない場合は記録のみ）
  - **更新成否を検証し、失敗時はユーザーに警告と再試行方法を提示する**
  - **完了時の次タスク導線（クイック報告）**:
    - 完了が成功し、残りtodoがある場合は「次のタスク」を1件提示する
    - 次を進めるために `開始 <番号|taskId>` を案内し、開始を `sessions.task_started` として記録する（以降の `完了/未達` はこのタスクに紐づく）
- `/api/jobs/weekly` (GET/POST): 直近7日ログから週次レビュー（DeepSeek JSON）を生成してPush
  - **モチベーション向上**: ポジティブなフィードバック（「今週の成果」「次の目標」「来週の焦点」）
- GitHub Actions: `.github/workflows/jobs-cadence.yml`, `.github/workflows/jobs-weekly.yml` で定期実行（Vercel Cronは無効）

## A (Architecture / Assemble)
### 実行基盤
- Next.js App Router（`app/api/**/route.ts`）が実運用の中心。
- Python側（`apps/api`, `apps/orchestrator`）は同居しているが、現行のLINE/タスク/ログの主フローはNext.js側。

### 外部連携
- **LINE Messaging API**: `replyMessage` / `pushMessage`
- **DeepSeek**: `https://api.deepseek.com/v1/chat/completions`（modelは `DEEPSEEK_MODEL` or `deepseek-chat`）
- **Google Sheets**: 永続化（goals/tasks/logs/sessions シート）

### API（Next.js）
- `POST /api/line/webhook`: LINE受信（message/postback）→ セッション制御/DeepSeek応答/タスク生成/日報更新/ボタンアクション
  - `x-line-signature` を検証し、不正な場合は 401 を返す
- `POST /api/line/postback`: postback処理ロジック（Webhook内のpostback処理と同等。テスト/分離のために残す）
  - 対応アクション（postback data）:
    - `approve_goal:<内容>`
    - `action=start_task&taskId=<ID>`
    - `action=snooze_task&taskId=<ID>`
    - `action=change_task`
- `POST /api/line/push`: 管理用 push（userId省略時は `LINE_USER_ID`）
  - `INTERNAL_API_KEY` による内部認証が必要（Authorization Bearer / `x-internal-api-key` / `?key=`）
- `GET /api/test-deepseek`: DeepSeek疎通確認用（デバッグ用途）

### 1. キーワードレス化
**概要**: ユーザーが `#整理開始` などのキーワードを覚えなくても、番号やAIの自動判断でモードを開始できる。

**実装内容:**

- **入口の統一（ボタン優先）**: アクティブセッションが無い状態で自由入力が来た場合は、まず「何をする？」のメニュー（ボタン）を提示し、ユーザーの選択でモードを開始する（自由入力での自動開始はしない）
- **「終了」で終了**: `#整理終了` の代わりに「終了」でも終了できる
- **LINEクイックリプライボタン**: タップで選択できる

**判断キーワード:**
- 思考ログ: モヤモヤ、悩み、考え、迷、不安、困、どうしよう、わからない
- 日報: 報告、完了、未達、done、miss、やった、できた、できなかった

### 7. ステータス確認（#ステータス）
**概要**: 現在の設定、タスク、ゴール進捗、統計情報を番号選択式で確認できる。

**コマンド:** `#ステータス` または `#状態` または `#status` または `status` または `ステータス`

**動作フロー:**
1. コマンド送信 → ステータスモード開始
2. 番号選択メニューを表示（1〜8、0=終了）
3. ユーザーが番号を送信 → 選択した項目の詳細を表示
4. 続けて他の番号を選択可能
5. `0` または `終了` でステータスモード終了

**コマンドの優先順位:**
- **常に優先されるコマンド**: どのモード（思考ログ、日報、ステータス）中でも以下のコマンドは常に優先して実行される
  - ヘルプコマンド: `/help`, `?`, `#ヘルプ`
  - リセットコマンド: `#リセット`, `reset`
  - **設定コマンド**: `#設定 キャラクター 社長`, `#設定 トーン 敬語`, `#設定 名前 田中`
  - **タスク関連コマンド**: `status <taskId>`, `split <taskId>`, `retry <taskId>`
  - **ゴール関連コマンド**: `#ゴール一覧`, `#ゴール進捗`, `#ゴール完了 <ゴール名>`
- これにより、ステータスモード中に `#設定` や `status t_123` などを送っても、「1〜8の番号、または 0（終了）を送ってください。」というエラーは出ず、正しくコマンドが実行される

**メニュー項目:**
- **1️⃣ パーソナライズ設定** - キャラクター、トーン、表示名、変更方法
- **2️⃣ 今日のタスク** - 朝の命令タスク、重要なタスク（優先度A、期限が近いもの）
- **3️⃣ ゴール進捗** - アクティブなゴール（最大5件）、進捗バー、完了件数
- **4️⃣ タスクサマリー** - 残りタスク総数、優先度別（A/B/C）、期限切れ
- **5️⃣ 最近の活動** - 連続日数（ストリーク）🔥、直近3日の記録、完了タスク
- **6️⃣ 統計情報** - 今週・今月の完了件数、全体の完了率
- **7️⃣ 推奨アクション** - 状況に応じた次のアクション提案（最大3件）
- **8️⃣ すべて表示** - 全項目を一度に表示
- **0️⃣ 終了** - ステータス確認を終了

**セッション管理:**
- ステータスモード中は他のモード（思考ログ、日報）を開始できない
- `0` または `終了` でモードを終了する必要がある
- `#リセット` で強制終了も可能

**表示内容詳細:**
1. **パーソナライズ設定**
   - キャラクターロール（社長/御曹司/アスリート/研究者/デフォルト）
   - メッセージトーン（厳格/敬語/フレンドリー）
   - 表示名（カスタム設定）
   - 変更方法の案内（`#設定 キャラクター 社長`など）

2. **今日のタスク**
   - いま進めているタスク（直近の `task_started` / `morning_order` から推定、todoのみ）
   - 朝の命令タスク（今日の焦点）
     - 朝の命令タスクが `done/miss` の場合は「完了済み」と表示し、完了済みタスクの詳細（説明/ID）は出さない
   - 朝の命令が完了済み/未設定で、かつ「いま進めている」が無い場合は、次に進める **おすすめタスクを1件** 表示する
   - 重要なタスク（優先度A、期限が近いもの最大3件）
   - **意思決定ボタン（Flex）**:
     - 「いま進めている」: `✅ 完了`（メッセージ送信: `完了`） / `❌ 未達`（`未達 `） / `🔄 変更`（postback: `action=change_task`）
     - 「朝の命令（todo）」/「次のおすすめ」: `▶ 今すぐ開始`（postback: `action=start_task&taskId=...`） / `⏰ 後で`（postback: `action=snooze_task&taskId=...`） / `🔄 変更`

#### ワンタップ確認の徹底（自由記述を減らす）
- Given: ユーザーが「次に何をすればいいか」を迷いやすい
- When: エージェントが「確定が必要な操作（開始/完了/未達/変更/取り消し）」を促す
- Then: **必ずボタン（ワンタップ）で確定できる導線**を提示する（テキスト入力は例外に限定）
- And: 自由記述が必要な場合（未達理由など）は「選択肢 + その他（自由入力）」を基本とする
- And: 自由記述は **必ず入口ボタン（例:「📝 その他（入力）」）の後にのみ受理**し、別操作が行われた場合は入力待ちをキャンセルする
- And: 自由記述が必要になった場合は、**必ず先に「📝 メッセージを入力してください」等の案内を出してから**、次のユーザー入力を受理する（いきなり自由入力を要求しない）
- And: 自由記述の内容を反映するような操作（設定変更/制約上書き/記憶の削除等）は、**入力→確認ボタン（✅ 保存/✅ 適用/✅ 削除 等）**の順で確定する（入力だけで自動確定しない）
  - 例: 「📝 メモする」→ 1行入力 → 保存（それ以外の雑談入力はメニューに戻す）
#### 1) コマンドレス「開始したい」→ 必ず確認付き（安全優先）
- Given: アクティブなセッション（思考ログ/日報/ステータス）が無い
- When: ユーザーが「今からやる/始める/着手する」等の自然文で“開始意図”を示す
- Then: エージェントは **開始対象のタスクを提示**し、**自動で開始記録はしない**
- And: 提示は Flex（`▶ 今すぐ開始` / `⏰ 後で` / `🔄 変更`）で、ユーザーのボタン操作をもって確定する

#### 2) 現在タスクのポインタ統一（current task）
- Given: 朝の命令提示/タスク開始/朝タスク変更等で「いまの焦点」が決まる
- When: ポインタが更新される
- Then: `sessions` に **current task の単一ポインタ**が記録される（クイック報告/ステータス/次タスク提示の共通参照）
- And: 既存の `morning_order` は cadence 抑制や当日ベースライン用途として保持しつつ、表示・報告は current task を優先する

#### 3) 開始時の分割提案（任意・確認付き）
- Given: ユーザーがタスクを開始した
- When: タスクが重そう/長そう（ヒューリスティクス）な場合
- Then: エージェントは `split <taskId>` の導線（提案）を提示する
- And: 分割の確定（サブタスク作成）は既存仕様どおり「承認」等の明示操作で行う

#### A) コマンドレス「完了/未達っぽい発話」→ 必ず確認付き（安全優先）
- Given: アクティブセッションが無い
- When: ユーザーが「終わった/できた/片付いた」等の自然文で“完了意図”を示す（ただし `完了` 単体などの確定コマンドは除く）
- Then: エージェントは「どのタスクを完了扱いにするか」を提示し、**ボタンで確認**させる
- And: ボタンは `✅ 完了として記録`（メッセージ送信: `完了`） / `❌ 未達として記録`（`未達 `）を提供する

#### 救済: undo（誤爆の取り消し）
- Given: 直近に tasks の status を変更した（done/miss 等）
- When: ユーザーが `undo` / `取り消し` / `元に戻す` を送る
- Then: 直近の変更を **1件だけ**取り消して、status を元に戻す
- And: 取り消しできない場合（対象無し/期限切れ/既に別状態）には理由を明示する

#### B) 「変更」の条件指定をボタン化（スマホのみ/軽い/休む）
- Given: ユーザーがタスク選択を変えたい
- When: ステータス（今日のタスク）等のFlex上で条件ボタンを押す
- Then: エージェントは条件（例: `スマホのみ` / `軽い` / `今日は休む`）に応じた候補提示またはスキップを実行する

#### C) 次タスク候補の決定ロジック（LLM無しの確定ルール）
- Given: 「次のおすすめ」を表示する必要がある（朝の命令が完了済み/未設定、かつ current task が無い）
- When: 次タスク候補を選ぶ
- Then: まず直近の進捗傾向を見て **miss傾向が強い場合は“軽め”を優先**する
- And: 次に **ゴール停滞（直近のdoneが古いゴール）** のタスクを優先する
- And: 次に **期限が近い** タスクを優先する
- And: 次に **優先度（A→B→C→不明）** を優先する

#### D) 未反応（黙っている）の扱いを統一（押しすぎ防止）
- Given: ユーザーの未反応は「忙しい/面倒/寝てる」を含む
- When: エージェントが「確定待ちの提案（開始/完了/未達/日次見直し適用 等）」を送った後、未反応が続く
- Then: リマインドは **最大2回/日** までに制限し、それ以上は当日停止する
- And: 当日停止中は、翌日に「要約 + 入口ボタン（ステータス/一覧）」のみを提示して再開できるようにする

#### E) 「あとで（スヌーズ）」を標準導線化（missより軽い逃げ道）
- Given: いま進める候補が提示されている
- When: ユーザーが `⏰ 後で`（スヌーズ）を押す
- Then: `sessions.task_snoozed` を記録し、**同一タスクは当日中の自動提案（次のおすすめ/朝の焦点の変更候補など）から除外**する
- And: スヌーズの再通知は「30分/1時間/2時間/今日スキップ」から選べ、確定はボタンで行う
- And: 今日スキップの場合は当日中は再通知しない（翌日以降は通常どおり候補に戻る）

#### F) 着手後の「時間ブロック」（任意・ボタンで選択）
- Given: ユーザーがタスク開始を確定した（postback start_task）
- When: エージェントが集中ブロックの提案を提示する
- Then: `25分/50分/90分/なし` のボタンを提示し、押下された場合のみ「タイマー終了通知」を送る
- And: タイマーは1回だけ通知し、同日に複数タイマーを乱立させない（上書きする場合は直近のみ有効）

#### G) タスク品質ガード（曖昧/大きい → 分割/相談を強制的に併設）
- Given: タスクが提示される（朝の焦点/次のおすすめ/ステータスの今日のタスク）
- When: タスクが「大きい/曖昧」だと判定される（例: 文字数が長い、`検討/調べる/考える/整理` 等の曖昧動詞）
- Then: カードのアクションに **必ず** `🧩 split` と `🛠 相談` を併設する（決定はボタン）

#### H) 日次見直し提案の衝突解決（最新優先）
- Given: 日次見直し（daily-review）の提案が存在する
- When: 同日に新しい日次見直し提案が生成される
- Then: **最新の提案のみを有効**とし、古い提案は superseded として扱い、適用ボタンは無効（または無視）にする
- And: ユーザーには「提案が更新された」ことを明記する（混乱防止）

#### I) 適用/更新の返信に変更点サマリを付与（undoの安心感）
- Given: タスク更新が行われる（done/miss/undo/日次見直し適用/相談からの優先度/期限/ゴール/文言変更 等）
- When: 更新が成功した
- Then: 返信には「変更点（before→after）」を1行で含め、`↩ undo` を併設する（可能な範囲で）

#### J) 「今日はここまで」（当日停止・ボタン）
- Given: ユーザーが当日の進行を止めたい（終業/就寝/これ以上やらない）
- When: ユーザーが `🌙 今日はここまで` を押す
- Then: 当日中の nudge/タイマー終了/スヌーズ再通知などの自動通知を停止する（夜の確認は別途）
- And: 翌日朝に通常運用へ戻る（自動解除）

#### K) 返信の「揺らぎ」ルール（seed + 直近重複回避）
- Given: 同じ文面が続くと味気ない
- When: エージェントが短文の定型返信（挨拶/励まし/促し/軽い相槌）を返す
- Then: 返信には「言い回しの揺らぎ」を入れる（同じ意味・同じ機能のまま言い方だけ変える）
- And: 揺らぎの選択は **seed付き**（例: `userId + kind + replyToken/planId + 時刻bucket`）で決める
- And: **直近N回（推奨: 3回）と同じvariantは避ける**（候補が1つしかない場合は除く）
- And: 採用したvariantは `sessions` に記録し、直近重複回避に使う
- And: morning/nudge/night/daily-review/timers 等の **ジョブ通知**も同じルールで「冒頭の一言」を揺らがせてよい（通知の意味は変えない）

#### L) Sayloぽさ（曖昧入力→意図候補提示→ボタン確定）
- Given: ユーザーが自由入力で話しかけるが、何をしたいかが曖昧なことがある
- When: アクティブセッションが無い状態で、コマンドに該当しない自由入力が届く

- And: 「どれでもない」用に `↩ メニュー` / `📊 ステータス` / `📋 一覧（LIFF）` の入口は常に残す

3. **ゴールと進捗**

   - 進捗バー（視覚的表示: `█████░░░░░ 50%`）

### 2. タスク分割
**概要**: 大きなタスクをAIが細かいサブタスクに分割する。

- 各サブタスクに `description`, `priority`, `reason`（なぜこのサブタスクが必要か）を付与

### 3. タスク再挑戦
**概要**: `miss` タスクを `todo` に戻す。

**コマンド:** `retry 1` または `retry t_123`

**フロー:**
1. ユーザーが `retry` コマンドを送信
2. `miss` タスクを探す
3. ステータスを `miss` → `todo` に更新
4. 「もう一度やってみよう。今度はできる。」と励ます

### 4. タスク生成時の理由説明
**概要**: タスクが生成されるとき、「なぜこのタスクが必要か」を説明する。

**実装:**
- AI（DeepSeek）のプロンプトに `reason` フィールドを追加
- タスク一覧表示時に理由も表示

**データモデル:**
- `Task` に `reason?: string` を追加

### 5. 週次進捗の比較（先週比）
**概要**: 週次レビュー時に、今週と先週の記録件数を比較し、増減を表示する。

**実装:**
- 先週の同期間のログを取得し、件数を比較
- 増減率を計算し、📈（増加）📉（減少）📊（同じ）で表示

**例:**
```
今週の記録: 15件
📈 先週より3件多い（+25%）
```

### 7. 月次・四半期レビュー（週次cronで自動判定）
**概要**: 週次cronの実行時に、月末・四半期末を判定して追加でレビューを実行する。

**実装:**
- `isEndOfMonth()`: 翌日が別の月かを判定
- `isEndOfQuarter()`: 翌日が別の四半期かを判定（3月末、6月末、9月末、12月末）
- 月末の場合: 30日分のログとタスク実績を集計し、月次レビューを送信
- 四半期末の場合: 90日分のログとタスク実績を集計し、四半期レビューを送信

**月次レビューの内容:**
- 今月の記録件数
- 今月のタスク実績（完了件数、達成率）
- 総評（3〜5行、成長の視点）
- 今月の主な成果（3つ）
- 来月の目標
- 来月の最初の焦点タスク

**四半期レビューの内容:**
- 今四半期の記録件数
- 今四半期のタスク実績（完了件数、達成率）
- 総評（5〜7行、長期的な視点）
- 今四半期の主な成果（インパクトの大きいもの3つ）
- 来四半期の戦略的目標
- 来四半期の最初の焦点

**cronの制約:**
- Vercelのcronは2つまで（morning, weekly）
- 月次・四半期は週次cronに統合して実行

### 8. 学習とパーソナライズ
**概要**: ユーザーの行動パターンを分析し、AIが自動で提案を生成する。

**分析内容:**
- **優先度別の傾向**: A/B/Cごとの完了率
- **曜日別の傾向**: 0=日曜〜6=土曜ごとの完了率
- **missタスクのパターン**: よくmissするキーワード、平均文字数
- **doneタスクのパターン**: よく完了するキーワード、平均文字数
- **全体の傾向**: 総タスク数、完了数、未達数、達成率

**提案の種類:**
1. **優先度調整**: 優先度Aの達成率が50%未満 → 「Aタスクを減らすか、優先度を見直しましょう」
2. **タスク簡略化**: missタスクの平均文字数がdoneタスクの1.5倍以上 → 「タスクを細かく分割してみましょう」
3. **時間調整**: 特定の曜日の達成率が30%未満 → 「この曜日は負担の少ないタスクを割り当てましょう」
4. **励まし**: 全体の達成率が70%以上 → 「この調子でいきましょう」

**表示タイミング:**
- 日報終了時に最大2件の提案を表示

### フェーズ4: 予測と最適化（2024-12-30完了）

#### ゴール達成予測
**概要**: 過去4週間のタスク完了ペースから、ゴールの達成予定日を予測します。

**予測内容**:
- 完了予定日・週あたりペース・信頼度（高/中/低）・推奨アクション

**表示場所**:
- `#ゴール進捗 <ゴール名>` コマンド
- 週次レビュー

#### タスク配分最適化
**概要**: 全ゴールを分析し、優先順位付けします。

**優先度**:
- 🚨 urgent: 完全に停滞（2週間以上進捗なし）
- ⚡ high: 進捗が遅い、またはもうすぐ完了
- 📌 normal: 順調

#### リスク検知
**概要**: ゴールの状態を監視し、4種類のリスクを自動検知します。

**リスク**:
1. 停滞（2週間以上進捗なし）
2. 期限リスク（完了まで半年以上）
3. タスク過多（未完了20個以上）
4. ゴール間不均衡（進捗のバラツキ）

**実装**: `lib/core/goal-prediction-service.ts`
