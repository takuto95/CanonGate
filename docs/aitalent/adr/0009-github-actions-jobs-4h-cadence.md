# ADR 0009: GitHub Actionsでジョブ運用（4時間枠・条件付き送信・無料枠ガードレール）

- **Status**: Accepted
- **Date**: 2026-01-11
- **Deciders**: Product / Engineering
- **Related**:
  - `docs/adr/0003-protect-job-endpoints.md`（/api/jobs/* 保護方針）
  - `docs/bmad/spec-current.md`（morning/night/weekly の存在、Vercel Cron制約）
  - `docs/bmad/specs/08-jobs-and-notifications.md`
  - `docs/bmad/gaps.md`（時刻/タイムゾーン、DeepSeek失敗時、ジョブ認証）

## Context
現行は Vercel Cron により定期実行を行っているが、運用上の制約によりジョブ本数・柔軟な頻度設計が難しい。
プロダクト方針として「ゆいがエージェントとして能動的に呼びかける」頻度を増やしたいが、以下を同時に満たす必要がある。

- しつこくならない（ユーザー体験）
- 無料枠に収まる（GitHub Actions / 外部APIコスト）
- 乱用実行を防ぐ（/api/jobs/* の保護）
- 実装が壊れにくい（冪等性、失敗時の再試行方針）

本ADRでは、**“4時間に1回まで”** を上限としてジョブ運用を設計する。

## Decision
以下を採用する（提案）。

1) **スケジュールは「4時間枠」で最大6回/日実行する（ただし送信は条件付き）**
- 4時間枠ごとにワークフローは起動してよい。
- ただし、ユーザーへの Push は **最大1通/枠**、かつ「送る価値がある時のみ送る（0通も可）」とする。
- 具体スロットは **JST基準**で固定する（例: 08:00 / 12:00 / 16:00 / 20:00 / 00:00 / 04:00）。
- 深夜枠（04:00）は原則「送らない」（例外条件のみ）。

2) **送信コスト（LLM）を守るため、定期呼びかけは“原則LLMを呼ばない”**
- 呼びかけ本文はテンプレ＋既存ロジック（タスク選定/状況）で組み立てる。
- LLM（DeepSeek）は「ユーザーから返信が来て会話が発生した時」や「1日N回まで」など、別途上限を設ける。

3) **間引き（送らない条件）を第一級要件にする**
最低限、以下は送らない。
- **朝の通知（morning相当）以降にユーザーから返信がある**（=その日は関与できている、とみなす）
- todoが無い
- snooze中（スヌーズがある場合）

4) **送る条件は優先度順に決め、常に“1通だけ”にまとめる**
例（優先度順）:
- 期限当日/期限接近の優先度Aタスクが未着手
- `morning_order` が未完了
- missが続いていて、負荷軽減（分割/次の一手）を促したい
- 夜枠のみ: ストリークが切れそう など

5) **ジョブ実行はGitHub Actionsから、保護されたAPIを叩く形に寄せる**
- `/api/jobs/*` は `INTERNAL_API_KEY` 相当で保護する（具体方式はADR 0003で確定する）。
- GitHub Actions Secretsにキーを保持し、ヘッダ付与で呼び出す（URLクエリにキーは置かない）。

## Options Considered
- Option A: Vercel Cronのまま（本数制約の中でやりくり）
  - 長所: 部品が増えない
  - 短所: 4時間枠など柔軟な設計が難しい/制約が強い

- Option B: GitHub Actionsで4時間ごとに“必ずPush”
  - 長所: 実装が単純
  - 短所: しつこい/無料枠と外部APIコストが増えやすい

- Option C: GitHub Actionsで4時間枠起動し、条件付きでPush（本ADRの提案）
  - 長所: 能動性と節約の両立が可能
  - 短所: 判定ロジックが必要（ただし既存データで十分実装可能）

## Consequences
- **Positive**:
  - 「最大でも4時間に1回」の上限を守りながら、必要時にはきめ細かく呼びかけられる
  - 定期ジョブでLLMを呼ばない設計により、無料枠/コストが読みやすくなる
  - /api/jobs保護（ADR 0003）と一体で、乱用実行リスクを下げられる

- **Negative / Risks**:
  - 間引きが強すぎると「能動性が弱い」と感じる可能性
  - 間引きが弱すぎると「しつこい」と感じる可能性
  - タイムゾーン/日付跨ぎ（JST/UTC）で判定がブレるリスク（`gaps.md`の協議対象）

- **Operational notes**:
  - 初期は「送らない」寄り（保守的）に設定し、反応と負担を見て緩める
  - 失敗時の再試行は“二重送信”を避ける（冪等キー/送信済み記録が必要）
